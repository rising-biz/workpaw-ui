# Warm Pool + Name Decoupling Implementation Plan (Plan 4)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Execute task-by-task with TDD.

**Goal:** 把首次登录冷启动从 ~100s（app boot）压到 **<10s**——warm pool 预建 Running+scope=all 通用配置已推的实例，领用后只补推 scope=user 专属配置（毫秒级 label patch + 秒级 config push）。同时解耦实例 name 与用户身份（改用 uuid），使领用不改 name/path/ingress。

**Architecture:**
- **operator** 侧（池维护 + name 解耦）：CR Spec 新增 `pooled`/`assigned_user`/`enterprise_id`/`instance_id`（复用来 QwenPawInstance，不新增 CRD——spec §15 ④）；name helpers 从 `{email前缀}` 改为 `qwenpaw-{instanceID}`；ingress path 从 `/{username}` 改为 `/i/{instanceID}`；replenisher reconciler 每企业维持 `[min, max]` 个 `pooled=true && assigned_user=""` 且 phase=`warming` 的预建实例。
- **control-plane** 侧（领用 + 映射）：新 PostgreSQL 表 `instance_assignments`（`user_id, instance_id, enterprise_id, assigned_at, state`）；`ActivateInstance` 改造为「先查 assignment → 未命中则 `SELECT ... FOR UPDATE SKIP LOCKED` 领用 pooled 实例 → 补推 scope=user 配置 → 返回 connect」；pool 耗尽降级冷启动，前端收 `pool_exhausted`（§5.3）。Stop 保留 PVC、不回池。旧 email 实例懒迁移（共存）。

**Tech Stack:** Go, controller-runtime, GORM+Postgres, workpawv1alpha1 CRD, Viper config.

**Spec:** `docs/superpowers/specs/2026-07-05-first-login-wait-ux-design.md` §7 / §9 / §10 / §11 / §13 / §15。

## Global Constraints

- **不新增 CRD**：pool 状态通过 `QwenPawInstance.Spec` 新字段表达（spec §15 ④）。
- **writer 边界不变**：phase 唯一 writer 仍是 operator（new phases: `provisioning`/`warming`/`assigned`）；`config-state` 仍由 control-plane ConfigSyncReconciler 写；Pool 实例的 `scope=all` 通用配置由 ConfigReconciler 预推，`scope=user` 在领用后补推。
- **name 解耦懒迁移**（§11）：存量 email-prefix 实例保持原 name/path；新实例全部走 `qwenpaw-{instanceID}` + `/i/{instanceID}`；ingress 新老路径共存（nginx regex 无冲突）。
- **podTemplateEqual 陷阱（Plan 3 教训）**：任何新增 K8s 结构体比较用 `field-by-field`，禁用 `DeepEqual`（防止 K8s 默认填充死循环）。
- **ctx 变量 shadow（Plan 3 教训）**：router.Setup 中新增 ctx 参数的代码不得 shadow `Setup` 的 ctx 形参。

## Tasks

### Task 1: operator CR 新字段
**Files:** `api/v1alpha1/qwenpawinstance_types.go`
**目标:** QwenPawInstanceSpec 加 4 字段（pooled bool, assigned_user string, enterprise_id string, instance_id string）+ 加 `// +optional` kubebuilder 标记。QwenPawInstanceStatus 加 `config_state` 短字段（control-plane 的 `config-state` annotation 镜像到 status，方便 operator 读取）。跑 `make manifests` 重生成 CRD YAML + deepcopy。
**产出:** 新 CRD YAML 含 pooled/assigned_user/enterprise_id/instance_id 字段。

### Task 2: operator name 解耦 + ingress path
**Files:** `internal/controller/qwenpawinstance_controller.go`
**目标:** `statefulSetName`/`secretName`/`serviceName`/`ingressName` 全部从 `username` 改为 `instanceID`（从 CR Spec 读，向后兼容无 instanceID 的旧 CR——回退 username）。`ensureIngress` path 从 `/{username}` 改为 `/i/{instanceID}`（rewrite-target 改 `/$2`——原来 `/([^/]+)(/|$)(.*)` strip 第一段）。旧 CR 无 instanceID→保留旧 path。
**关键:** 所有 helper 保持兼容旧 email-prefix 实例（无 `Spec.InstanceID` → 回退 `name` 或 `instanceName(email)`）。

### Task 3: operator phase 扩展
**Files:** `internal/controller/phase.go` / `qwenpawinstance_controller.go`
**目标:** derivePhase 加入 `provisioning`（CR 创建后 Pod 未创建）和 `warming`（Pod Running + scope=all 配置已推 + pooled=true）。`warming` 判定：Pod Ready + Spec.Pooled==true + ConfigReconciler 已推完 scope=all（annotation `config-state` = `synced`）。

### Task 4: operator pool 维护雏形
**Files:** `internal/controller/qwenpawinstance_controller.go` / `pool.go`
**目标:** 加 `Config.PoolConfig`（`min_per_enterprise`, `max_per_enterprise`, `enabled`）读取已存在的 `workpaw-operator-config` ConfigMap（复用 CORS 读模式）。`replenish(ctx, enterpriseID)` 查询该企业 `pooled=true && assigned_user="" && DesiredState=Running` 的 CR 数；若 < min，通过 `r.Create` 异步补一个新 QwenPawInstance（name=`qwenpaw-{uuid}`，`Spec.InstanceID`=uuid，`Spec.Pooled=true`，`Spec.EnterpriseID`=enterpriseID）。并发受 `replenish_concurrency` 限制。
**v1 范围:** replenish 只在每次 reconcile Running 实例时**顺路检查**（不启动独立的 pool watcher）。若池不足，Create + 日志。下一轮 reconcile 继续维护。不做独立的池监控 goroutine。

### Task 8: control-plane pool 领用 flow（核心）
**Files:** `internal/service/instance.go`（改造 ActivateInstance）+ 新建 `internal/service/assignment.go`
**目标（实现 spec §7.4 完整领用算法）**:
1. `ActivateInstance` 改造入口：
   1. `disabled` guard（保持）。
   2. 查 `instance_assignments` WHERE user_id=? AND state='assigned' → 命中 → 读该 CR → 若 Stopped，DesiredState=Running 唤醒 → 返回 connect + `assignment=reusing`。
   3. 未命中 → 进入领用流程：
      - `SELECT instance_id FROM QwenPawInstance WHERE pooled=true AND assigned_user='' AND enterprise_id=? LIMIT 1 FOR UPDATE SKIP LOCKED`（防并发抢同一 pooled）
      - 命中: UPDATE CR label `assigned_user`=userID + 打 annotation `workpaw.dev/assigned-at`=now → INSERT `instance_assignments`(state=assigned) → 触发 §6.4 ConfigSyncReconciler 补推 scope=user 专属配置 → 返回 connect（`/i/{instanceID}`+token）
      - 未命中（pool 耗尽）: 降级——新建普通 CR（按 §6 冷启动路径，name=uuid-based），前端收 `assignment=cold`（→ §5.3 `pool_exhausted` 文案）。
2. `instanceName` → `generateInstanceID()` 返回 uuid 字符串（`strings.Replace(uuid.New().String(), "-", "", -1)` 的短形式或标准 UUID）。保留旧 `instanceName(email)` 在旧路径兼容使用。
3. `GetInstance`: 先查 `instance_assignments WHERE user_id=? AND state='assigned'` → 找到 → 读对应 CR。未找到 → 回退旧 email-prefix 查找（向下兼容旧 CR）。
4. `DeactivateInstance`: DesiredState=Stopped（PVC 保留——控制面不改 CR，仅改 DesiredState）。state 不变（仍 assigned）→ 下次 activate 走 assignment 命中 → 唤醒（快路径，不复冷启动）。**不回池**（CR label 不变 pooled=false、token 不变——security guarantee 见 spec §15 ③）。
5. `ConnectInfo`: 读 CR Secret → 取 api-token + ingress 信息。

### Task 9: control-plane ConfigReconciler scope=all 预推
**Files:** `internal/service/config_reconciler.go` / `internal/service/config_sync_controller.go`
**目标:** ConfigReconciler 在 `convergeUser` 中检测 CR `pooled=true && assigned_user=""` → 只推送 scope=all bindings 的 DesiredConfigs（scope=user 的 skip）。ConfigSyncReconciler 在新 `assigned_user` label 被 set 时额外触发一次收敛（补推 scope=user）。第 60s ticker 仍是 safety-net。
